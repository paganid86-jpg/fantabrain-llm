# Output Filter v0 Design

## Context

`qwen25-3b-fantabrain-sft-v2` remains the current champion adapter. The guarded v2 eval reduced some uncertainty but still produced hard failures, especially invented Mantra modules. The latest audit hotfix also proved that the text audit is useful, but it is still shaped as an offline report.

The next product step is to turn the audit behavior into an app-oriented output filter. The filter should decide whether a model answer is safe to show, should trigger a fallback model, or should be replaced by a conservative safe response after repeated failure.

## Goal

Add a reusable output filter that can classify a single FantaBrain AI answer into one of four decisions:

- `pass`: answer can be shown directly.
- `pass_with_warnings`: answer can be shown, but non-blocking issues were detected.
- `fallback`: answer has a hard violation and should be regenerated by a fallback model.
- `safe`: answer should not be shown and should be replaced by a conservative safe response.

The first consumer is the FantaBrain app integration path:

```text
User prompt
  -> Qwen/LoRA answer
  -> output filter
  -> pass: show Qwen answer
  -> fallback: call gpt-5.4-mini, then filter again
  -> safe: show conservative safe response
```

## Non-Goals

- Do not call OpenAI from this branch.
- Do not implement the FantaBrain app backend integration yet.
- Do not train a new adapter or create another dataset.
- Do not rewrite or post-process answers to hide model mistakes.
- Do not change Pagella v0 cases or scoring.

## Filter Behavior

The filter should reuse the existing deterministic prediction audit rules where possible. The audit remains the source of violation detection; the new filter adds routing semantics and a smaller single-answer API.

### Decisions

`pass`:

- No violations were detected.

`pass_with_warnings`:

- Only soft violations were detected.
- Examples: malformed known terms, Classic module-language warnings such as `slot` or `incastri`.
- Product behavior: show the answer for now, but log the warning.

`fallback`:

- At least one hard violation was detected.
- Examples: invented Mantra modules, Mantra `modificatore` leakage, Classic standalone Mantra role-code leakage.
- Product behavior: do not show the answer; call the fallback model.

`safe`:

- The answer is empty, whitespace-only, or structurally unusable.
- The fallback path can also ask for `safe` after a second failed filter pass.
- Product behavior: show a conservative response that names missing context and avoids invented specifics.

### Presets

The initial preset is `app_v0`.

`app_v0` should be strict on hard mode leakage and lenient on soft style issues. This keeps launch behavior safe without over-triggering the fallback for every imperfect sentence.

## Public API

Add `src/fantabrain_llm/output_filter.py`.

Suggested types:

```python
class FilterAction(str, Enum):
    PASS = "pass"
    PASS_WITH_WARNINGS = "pass_with_warnings"
    FALLBACK = "fallback"
    SAFE = "safe"


@dataclass(frozen=True)
class FilterDecision:
    action: FilterAction
    reason: str
    violations: list[AuditViolation]
```

Suggested function:

```python
def filter_model_output(
    *,
    mode: str,
    task: str,
    prompt: str,
    prediction: str,
    preset: str = "app_v0",
    fallback_failed: bool = False,
) -> FilterDecision:
    ...
```

`fallback_failed=True` means this is already the fallback model's second chance. If hard violations are still present, return `safe` instead of another `fallback`.

## CLI Shape

Add `scripts/filter_predictions.py` for offline validation of existing Pagella runs:

```bash
python scripts/filter_predictions.py \
  --predictions reports/runs/qwen25-3b-fantabrain-sft-v2-mode-fence-v1-pagella-v0/predictions.jsonl \
  --output-dir reports/runs/qwen25-3b-fantabrain-sft-v2-mode-fence-v1-pagella-v0
```

The CLI should write:

- `output_filter.json`
- `output_filter.md`

The JSON report should include:

- total cases;
- decision counts;
- violation counts;
- per-case action, reason, and violations.

The Markdown report should be short and readable for manual review.

## Safe Response Direction

This branch should only expose enough structure for a future app-safe response. It should not over-design a copy system.

Initial conservative response shape:

```text
Mi manca contesto sufficiente per chiuderla con sicurezza.
Posso darti una lettura prudente: evita scelte basate su moduli, ruoli o dati non confermati e confronta titolarita, copertura e rischio prima di decidere.
```

Future app integration can specialize the copy by mode and task.

## OpenAI Fallback Direction

The fallback provider is intentionally out of scope for this branch, but the filter API should make the next step straightforward.

Recommended next branch after this filter:

- add an optional OpenAI eval provider path;
- test `gpt-5.4-mini` as the real fallback candidate;
- optionally test `gpt-5.5` as a quality oracle on a small subset;
- run the same filter on fallback outputs before showing them.

`gpt-5.5` should not be the default production fallback because of cost. It is useful as a benchmark model only.

## Testing

Use TDD. Add tests before production code.

Required tests:

- clean Mantra output returns `pass`;
- malformed-only output returns `pass_with_warnings`;
- Mantra output with `modificatore` returns `fallback`;
- Mantra output with invented module returns `fallback`;
- Classic output with new standalone role code returns `fallback`;
- empty output returns `safe`;
- hard violation with `fallback_failed=True` returns `safe`;
- CLI writes JSON and Markdown reports for a small predictions file.

## Rollback

This is an additive filter layer. If the filter is too aggressive or too permissive, app integration can keep using the previous audit-only path while rules are adjusted.
